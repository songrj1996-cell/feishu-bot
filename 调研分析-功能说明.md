# /调研分析 功能说明

这份文档描述 `/调研分析` 这个指令从用户发送到拿到报告之间，整个系统是怎么运作的——每一步发生在哪个平台、由哪段代码负责、为什么要这么做。

读者目标：让不熟悉代码的同事能照着这份文档把整个流程脑补出来；也能在系统出问题时知道该看哪个环节。

---

## 0. 一句话概括

> 用户在飞书私聊里发 `/调研分析 <表格链接>`，机器人读完表格 → **planner Dify 应用**给出"列分类 + 报告章节 + 交叉分析"的方案 → **必出**一张确认卡片让用户审 → 用户回 OK → **Python 算所有统计数字** → **analyst Dify 应用**按 part 章节写报告 → 飞书文档发回（所有权转给用户）→ 用户可对报告消息「回复」继续追问。

⚠️ **2026-05 架构重构**：之前是把整张表 markdown 一把丢给 Dify、由 LLM 既算数字又写报告，结果数字经常算错。重构后**所有频数 / 占比 / 交叉表都由 Python 算**，LLM 只负责规划 + 撰稿 + 答疑。详见末尾「重构后 Dify 端配置」章节。

涉及到的平台：

| 平台 | 在这个流程里负责什么 |
| --- | --- |
| **飞书开放平台** | 接消息 / 发消息 / 读表格 / 创建文档 / 转移文档所有权 |
| **Dify**（公司内网部署） | 大模型推理：判断要不要追问、写报告 |
| **公司内网服务器** | 跑 `feishu-bot` 进程，作为飞书事件订阅的长连接客户端 |

## 1. 用户视角的两条路径

发完 `/调研分析 <link>` 后，**机器人接下来的动作只有两种可能**：

### 路径 A：直接出报告
1. 收到表格 → 读取数据 → 喂给 Dify
2. Dify 觉得信息够，直接生成完整报告
3. 机器人把报告写成飞书文档，回 `📄 报告生成完毕：<文档链接>`

### 路径 B：先澄清再出报告
1. 收到表格 → 读取数据 → 喂给 Dify
2. Dify 反问：「你这份调研主要关注哪类用户？」
3. 机器人把追问以**卡片**形式发出来
4. 用户用飞书的「**回复**」功能针对这条卡片作答
5. 机器人把回答送回 Dify（带上同一个 `conversation_id`，所以 Dify 还记得上下文）
6. Dify 这一轮直接出报告（也可能继续追问，回到 3）
7. 出报告后走和路径 A 一样的飞书文档流程

机器人靠一个简单启发式区分这两路：**Dify 返回的内容里有 markdown 标题（`# / ##`）或者长度超过 1500 字符 → 当成报告；否则当成澄清问题**。逻辑在 [main.py:_looks_like_report](main.py)。

为什么用「回复」功能而不是普通文字？因为同一个用户可能同时挂着好几份调研在分析（一份可能在等用户回答，另一份还在跑 Dify）。每条卡片都有自己的 `message_id`，用户用飞书"回复"那条卡片时，事件里 `parent_id` 会指向卡片的 `message_id`，机器人据此精确路由到正确的 Dify 会话。详见第 6 节。

## 3. 一步一步拆解

### 第 1 步：用户发送指令
- **平台**：飞书 IM
- **谁做**：用户

用户向机器人发送类似：

```
/调研分析 https://moontontech.feishu.cn/sheets/abc123?sheet=xxx
```

支持的链接类型：飞书表格（sheets）、飞书 wiki 节点（wiki）。后者会被自动解析到底层的 sheet。

### 第 2 步：飞书把消息推到机器人
- **平台**：飞书开放平台 → 服务器
- **谁做**：飞书 + lark-oapi 客户端

机器人不是 webhook，而是**长连接订阅**：进程启动时通过 lark-oapi SDK 连一条 WebSocket 到飞书，飞书有事件就直接推过来。这样我们不需要公网 IP / 域名 / HTTPS 证书——服务器只要能访问外网就行。

入口在 [main.py:438-451](main.py#L438-L451)：

```python
client = lark.ws.Client(
    FEISHU_APP_ID, FEISHU_APP_SECRET,
    event_handler=handler, log_level=lark.LogLevel.INFO,
)
client.start()  # 阻塞，长跑
```

注意 lark-oapi 的事件回调（`_on_message_received`）是**同步函数**，但我们的业务全是 async（要 await 飞书/Dify 的 HTTP API）。所以在 [main.py:411-419](main.py#L411-L419) 起了一个**后台线程跑 asyncio 事件循环**，同步回调里只负责把协程调度过去，立刻返回，避免阻塞飞书的事件分发。

事件去重：飞书可能因为 ack 超时重发，[main.py:_is_duplicate](main.py) 用 `event_id` 做 5 分钟内的去重。

### 第 3 步：解析消息 + 路由
- **平台**：服务器
- **谁做**：[main.py](main.py) + [commands.py](commands.py)

[main.py:_handle_message](main.py) 做这几件事：
1. 过滤掉非 p2p（群聊不响应）
2. 用 [feishu.py:get_user_name](feishu.py) 把发送者 open_id 换成姓名（用于日志）
3. 用 [commands.py:parse_command](commands.py) 识别 `/调研分析` 指令
4. 把指令分发到 [main.py:_handle_app_command](main.py)，根据 `COMMAND_TO_APP` 路由到 `analyze` 业务

`_handle_app_command` 里对应 `analyze` 分支：
1. 用 [feishu_sheets.py:parse_feishu_link](feishu_sheets.py) 校验链接格式
2. 立即给用户发一条**初始反馈**：「📊 收到表格，正在读取数据！如果有需要和你确认的问题，我会在 3 分钟内向你提问；如果没有收到我的提问，意味着我已经开始整理报告了哦~ 之后会每 5 分钟向你叭叭一句来表示我还在分析！」
3. 用 `asyncio.create_task` 把后续耗时工作丢到后台跑（飞书要求事件 3 秒内 ack，所以不能阻塞）

### 第 4 步：读取飞书表格 → markdown
- **平台**：飞书云文档 API → 服务器
- **谁做**：[feishu_sheets.py:fetch_as_markdown](feishu_sheets.py)

这一步是大头，分几个子步骤：

#### 4.1 解析链接
[feishu_sheets.py:parse_feishu_link](feishu_sheets.py) 用正则从 URL 抽出 `type`（sheets/wiki/...）、`token`、`sheet_id`（如果 URL 里带了 `?sheet=xxx`）。

#### 4.2 wiki 节点解析（如果是 wiki 链接）
飞书 wiki 节点本身只是个壳子，真正的 sheet 在底下。[feishu_sheets.py:resolve_wiki_node](feishu_sheets.py) 调 `wiki/v2/spaces/get_node` 拿到 `obj_token`，再继续按 sheet 处理。

#### 4.3 列出工作表 + 选定要读的那个
- 调 `sheets/v3/spreadsheets/{token}/sheets/query` 列出所有 sheet
- 如果链接里指定了 `sheet_id` → 选那一个
- 否则**只读第一个 sheet**（避免多 sheet 工作簿被全部拉下来。多 sheet 时会在日志里打出未读取的 sheet 名作为提示）

#### 4.4 读取数据
[feishu_sheets.py:read_sheet_values](feishu_sheets.py)：
- 先调 `get_sheet_meta` 拿到行数、列数
- 按 200 行一批分页读，调 `sheets/v2/spreadsheets/{token}/values/{range}`
- **遇到 code=90221（单批 >10MB）会自动把批次缩半再试**——飞书有 10MB/请求的硬限制
- 末尾全空行剔掉

#### 4.5 转 markdown 表格 + 加元信息
[feishu_sheets.py:values_to_markdown](feishu_sheets.py) 把二维数组转成标准 markdown 表格（管道 `|` 转义，换行转 `<br>`，富文本 cell 提取 text）。

外层 [feishu_sheets.py:fetch_as_markdown](feishu_sheets.py) 在表格前加一段 `<metadata>`：

```
<metadata>
sheet 名: xxx
总回复数: N 份（已读取，不含表头行）
列数: M
处理要求: 必须分析全部 N 份回复...
</metadata>
```

这段元信息让 LLM 知道**确切的样本量**，避免它"差不多分析一下就好"——很多时候 LLM 接到几百行数据会偷懒抽样。

### 第 5 步：调用 Dify「调研分析」应用
- **平台**：Dify（公司内网 `dify.web.moontontech.net`）
- **谁做**：[dify.py:chat](dify.py)

调研分析用的是 Dify 的 **chat-messages** 接口（对话式，带 `conversation_id`，可以多轮），不是 completion-messages。

请求长这样：

```python
POST {DIFY_API_BASE}/chat-messages
Authorization: Bearer {DIFY_API_KEY}
{
  "inputs": {},
  "query": "<sheet 转的 markdown>",
  "response_mode": "streaming",
  "user": "<飞书 open_id>",
  "conversation_id": ""  # 第一轮空，之后用 Dify 返回的 id
}
```

为什么用 streaming？两个原因：
1. 报告动辄上千字，非流式接口可能等很久才返回完整响应
2. Dify 流式响应里会逐 chunk 推 `data:` 事件，我们累加 `event=message` 的 answer 字段，在 `event=message_end` / `workflow_finished` / `agent_message_end` 时跳出循环

返回值：`(answer_str, new_conversation_id)`。后者是**多轮对话的关键**——澄清场景下要把它存起来，等用户回答时带回去。

### 第 6 步：心跳 + 5 分钟进度同步
- **平台**：服务器
- **谁做**：[main.py:_heartbeat](main.py) / [main.py:_with_heartbeat](main.py)

Dify 跑大模型可能要几十秒到几分钟。用户看不到任何动静会以为机器人挂了。所以等 Dify 期间会起一个心跳协程，每 5 分钟发一条「在写了在写了，别急别急！已等待 X 分钟」。

实现是个简单的 `asyncio.create_task` + `try/finally cancel` 模式——主协程跑完就把心跳 cancel 掉，避免泄漏。

### 第 7 步：分流——澄清问题 vs 报告
- **平台**：服务器
- **谁做**：[main.py:_send_analyze_answer](main.py) + [main.py:_looks_like_report](main.py)

启发式判断：
```python
def _looks_like_report(answer: str) -> bool:
    if re.search(r"^#{1,6}\s", answer, flags=re.MULTILINE):  # 有 markdown 标题
        return True
    return len(answer) >= 1500  # 没标题但很长
```

#### 7A 是报告 → 走飞书文档流程（见第 8 步）
#### 7B 是澄清 → 用卡片发回，登记 pending 状态

卡片发送由 [feishu.py:reply_markdown](feishu.py) → `_reply_card` 完成。返回值是**这条卡片的 `message_id`**，记下来。

然后调 [main.py:_add_pending](main.py) 登记：

```python
user_state[user_id][bot_msg_id] = {
    "conv_id": new_conv_id,           # Dify 会话 id，下一轮要带回去
    "app_id": "analyze",
    "sheet_names": [...],             # 后续清洗标题用
    "ts": time.time(),                # 24 小时 TTL
}
```

### 第 8 步（仅报告路径）：生成飞书文档
- **平台**：飞书云文档 + 飞书云盘 API
- **谁做**：[feishu_docs.py:create_doc_from_markdown](feishu_docs.py)

飞书没有"直接传 markdown 创建文档"的接口，得绕一圈：

#### 8.1 标题清洗
[main.py:_normalize_report_title](main.py) 提取首个 `#` 标题，剥掉 sheet 名（如「Sheet 1」「问卷-第 23 期」之类），让文档名更干净。如果没标题，用兜底「调研报告 YYYY-MM-DD HHMM」。

#### 8.2 上传 md 作为文件
[feishu_docs.py:_upload_md_file](feishu_docs.py) 调 `drive/v1/files/upload_all`：
```
multipart/form-data:
  file: <markdown bytes>
  file_name: <title>.md
  parent_type: explorer
  parent_node: ""    # 上传到机器人自己的根目录
```
返回 `file_token`。

#### 8.3 发起导入任务（md → docx）
[feishu_docs.py:_create_import_task](feishu_docs.py) 调 `drive/v1/import_tasks`：
```
{
  "file_extension": "md",
  "file_token": "<上一步的 file_token>",
  "type": "docx",
  "file_name": "<title>",
  "point": {"mount_type": 1, "mount_key": ""}
}
```
返回 `ticket`。

#### 8.4 轮询任务状态
[feishu_docs.py:_poll_import_task](feishu_docs.py) 每 1.5 秒查一次 `drive/v1/import_tasks/{ticket}`：

| `job_status` | 含义 |
| --- | --- |
| 0 | 成功 |
| 1, 2 | 处理中 |
| 3+ | 错误 |

> 这里有个**反直觉的坑**：飞书文档里没明说 0 是成功，最早写代码时按 "0=init" 理解，结果整个流程都没跑通。已经写在了 memory 和注释里。

成功后拿到 `token` + `url`。

#### 8.5 转移所有权给用户
[feishu_docs.py:_transfer_owner](feishu_docs.py) 调 `drive/v1/permissions/{doc_token}/members/transfer_owner`：
```
POST ?type=docx
{"member_type": "openid", "member_id": "<用户 open_id>"}
```

转移后文档会自动出现在用户的「我的空间」下。**机器人保留 full_access 权限**（飞书的转所有权语义），所以仍可以读这份文档（如果以后需要的话），但用户成了主人，可以自由删除/分享。

为什么要转所有权？最初的版本是给用户加协作者权限，结果文档显示在「共享空间」里，用户找不太到。转所有权后变成「我创建的」，体验明显更顺。

#### 8.6 把链接发给用户
回 `📄 报告生成完毕：\n<doc_url>`。结束。

### 第 9 步（仅澄清路径）：用户回答 → 续聊
- **平台**：飞书 → 服务器
- **谁做**：[main.py:_handle_message](main.py) + [main.py:_continue_conversation](main.py)

用户用飞书"回复"功能针对那条卡片回答时：
1. 飞书把消息推过来，里面 `parent_id = <那条卡片的 message_id>`
2. [main.py:_resolve_pending](main.py) 用 `(user_id, parent_id)` 精确匹配到登记的 pending 状态
3. 如果用户**没**用回复功能（直接发文字），[main.py:_take_single_pending](main.py) 兜底——只有一个 pending 时直接用，多个时让用户去用回复功能
4. 命中 pending 后，调 [main.py:_continue_conversation](main.py)：
   - 立即发「📝 开始整理报告...」（澄清回答完通常下一步就是出报告，给用户心理预期）
   - 把用户回答 + 之前存的 `conv_id` 喂回 Dify
   - Dify 这一轮多半返回报告，走第 8 步生成飞书文档
   - 也可能继续追问，那就再次登记 pending，回到第 7B

## 5. 关键状态：pending 表

[main.py](main.py) 里维护：

```python
user_state: dict[str, dict[str, dict]] = {}
# user_id → {bot_msg_id → {"conv_id", "app_id", "sheet_names", "ts"}}
```

**每条 pending 表示一个等用户回答的澄清问题**。键是机器人发出的卡片消息 id，值里包含让续聊能继续下去的所有上下文。

为什么要按 `bot_msg_id` 索引而不是按 `user_id` 一对一？因为同一个用户可能同时挂着好几个分析任务：

- 早上 9 点发了一个表 A，机器人问了澄清问题，用户没及时答
- 上午 10 点又发了一个表 B，机器人正在跑 Dify，还没出结果
- 10:05 用户回到表 A 那条卡片去回答了澄清问题

这时如果状态只按 `user_id` 存，新的回答可能被错误地路由到表 B 的会话上。靠 `bot_msg_id + parent_id` 精确匹配能避开这个坑。

**TTL**：24 小时。每次访问 pending 时顺便清过期项，简单但够用。

## 6. 错误 / 重试 / 限流

| 场景 | 处理 |
| --- | --- |
| 飞书事件重试（同一条消息推多次） | `processed_events` 按 `event_id` 5 分钟去重 |
| 飞书 `tenant_access_token` 过期 | [feishu.py](feishu.py) 自带带过期时间的 dict 缓存，提前 60s 续 |
| Dify chat-messages 流式断开 / 报错 | 抛异常 → 顶层 `_run_analyze` 捕获 → 给用户回错误消息 + traceback 进日志 |
| Dify 接口 400（变量类型错） | `dify.complete()` 返回 STOP_SIGNAL（这个针对打标，调研分析用 `dify.chat()` 直接抛错） |
| 飞书 sheet 单批 >10MB（code 90221） | [feishu_sheets.py:read_sheet_values](feishu_sheets.py) 自动把批次缩半再试 |
| 飞书 import 任务失败 | `_poll_import_task` 抛 RuntimeError，外层兜底 |
| 长文本超 25000 字（飞书消息 30K 上限） | [feishu.py:_split_text](feishu.py) 按 `## / # / 段落 / 行` 优先级裁切 |

特别说明：调研分析这条线**不会**插入列、改写表格——它只读，所以风险面比 `/反馈打标` 小很多。最坏情况就是给用户回个错误消息，原表格毫发无损。

## 9. 为什么这么设计——几个关键决定

### 9.1 长连接而不是 webhook
公司内网服务器没有公网 IP / 域名 / HTTPS。webhook 路线要么开公网映射（运维成本 + 安全审查），要么用 ngrok 这类隧道（不能跑生产）。lark-oapi 提供的长连接订阅完美绕过这个问题——服务器只要能**主动出网**就行。

### 9.2 卡片 + reply parent_id 路由
最直白的实现是按 `user_id` 存最近一次会话状态。但同一个用户可能并发发多份调研，状态会互相覆盖。用卡片 message_id 作为路由键 + 飞书"回复"功能的 parent_id，能天然支持每用户多会话并发。

### 9.3 文档而不是卡片返回报告
飞书消息有 30K 字符上限，卡片渲染长 markdown 也很别扭（不支持 #-标题、表格断行）。文档形式既无长度限制，也方便用户后续编辑、分享。

### 9.4 转所有权而不是加协作者
所有权转移后文档进用户「我的空间」，找得到、删得掉。加协作者那条路最初试过，用户反馈"在共享文档里翻了半天"。

### 9.5 分流的启发式（标题 OR 长度）
依赖 Dify prompt 教大模型"出报告时用 # 标题"是最干净的。但万一 Dify 偶尔不带标题，长度兜底能避免一份完整报告被错当成澄清问题塞进卡片。

### 9.6 metadata 块塞进 prompt
LLM 接到几百行表格数据有偷懒倾向（"我看了一下大概是..."）。明确告诉它"必须分析全部 N 份回复，要排除的请明确列出"能逼它老实做完。

## 11. 重构后 Dify 端配置（必读）

> 2026-05 重构把单 Dify 应用拆成了两个 chat 应用。如果你是首次部署或者刚 pull 了重构后的代码，**必须**在 Dify 后台新建这两个应用、贴上下面的 prompt、把 API Key 填到 .env 才能跑通。

### 11.1 需要建几个应用？

两个 **Chat 应用**（不是 Workflow，不是 Completion）：

| 应用名建议 | 角色 | API Key 写到 .env 哪个变量 |
|---|---|---|
| 调研分析-规划器 | 看表头+前 5 行样本，输出列分类 / part 划分 / 交叉建议的 JSON | `DIFY_PLANNER_KEY` |
| 调研分析-分析师 | 同一会话先按 part 章节写报告，后续 QA 用同一 conv_id 续聊 | `DIFY_ANALYST_KEY` |

**不要**复用旧的「调研分析」应用——重构后的 prompt 跟旧 prompt 完全不兼容（旧 prompt 同时教 LLM 算数字 + 写报告，新 prompt 把这两件事拆开了）。

### 11.2 模型选择建议

- 规划器：用 Claude Sonnet 4 / GPT-4o 等中等模型即可，输入小（表头 + 5 行）输出小（一段 JSON）
- 分析师：建议用**长上下文**模型（Claude Sonnet 4 200K / GPT-4 Turbo 128K），因为后续 QA 会在同一会话里累积 stats + 报告 + rows + 多轮问答

### 11.3 Planner（调研分析-规划器）prompt 模板

把下面整段贴到 Dify 应用的「提示词」/「编排」字段：

````
你是 MLBB 游戏用户调研问卷的「分析方案规划助手」。给定一份表头和前 15 行样本，输出一个 JSON（用 ```json 围栏包起来），不要附加多余解释。

JSON schema：
```json
{
  "columns": [
    {
      "index": 0,
      "name": "题目的中文短名",
      "role": "id | profile_dim | single_choice | multi_choice | scale | open_text | ignore",
      "confidence": "high | medium | low",
      "delimiter": "(仅 multi_choice，看样本判断 , / ， / ; / ； / 、 哪个)",
      "min": 1, "max": 10,
      "value_aliases": {
        "王者": ["王者", "Mythic", "Mítica"]
      }
    }
  ],
  "parts": [
    {"name": "章节名（中文）", "column_indexes": [0, 1, 2]}
  ],
  "open_questions": ["看不懂的列 / 不确定的事，想问用户的话"],
  "summary": "一句话描述这次分析方案 + 你是怎么划分 parts 的"
}
```

**name 字段规则**：

- `name` 必须是该列的**中文短名**（≤ 12 字），用于直接展示给用户
- 原始表头是中文 → 提炼成短名（如「你最常玩的位置是什么？」→「主玩位置」）
- 原始表头是英文 / 西语 / 葡语 / 任何其他语言 → 翻译并提炼成中文短名（如 "Which device do you play on?" →「使用设备」）
- **不允许把原文表头原样塞到 `name`**——用户看到英文 / 葡语会读不懂
- 提炼时保留题目的关键语义，不要丢失业务含义

**role 选择规则**：

- `id`：用户身份标识列（mlbbid、discord、whatsapp、邮箱）。不参与统计但 QA 阶段会用到。下游展示玩家信息按 mlbbid > discord > whatsapp 优先级
- `profile_dim`：**用户画像维度**——任何能用于人群分组分析的字段都属于画像，包括但不限于：
  - 段位（青铜 / 白银 / 黄金 / 铂金 / 钻石 / 星耀 / 王者 / 神话…）
  - 游戏年限 / 时长（玩了多久、每天玩几小时）
  - 付费层级（月卡用户、648 大额付费、零付费等）
  - 主玩位置 / 英雄类型（射手 / 法师 / 战士 / 刺客 / 辅助 / 坦克）
  - 地区（国家 / 城市 / 服务器）
  - 年龄段 / 性别 / 设备
  - 其他能区分玩家群体的字段
  - **不确定某列是不是画像 → 放到 `open_questions` 问用户，不要自作主张**
- `single_choice`：选项数有限的单选题
- `multi_choice`：多选题。**必须**给 `delimiter` 字段
- `scale`：1-N 的量表 / 评分题（NPS / 满意度 / 推荐意愿等）。**必须**给 `min` / `max`
- `open_text`：开放文本题
- `ignore`：时间戳、提交 ID 等不参与分析的列

**parts 划分规则（重点）**：

- **看题目间的前后关联**——如果一道选择题后跟着一道主观题让玩家解释自己选择的原因，把它们归到同一个 part。问卷常见这种"选择题 + 解释题"对子模式
- **基本信息 / 画像类列**归到第一个 part（例如"基本信息"）
- 每个 part 内可以同时含画像、客观题、开放题——**不要**按题型切分 part
- **请在 `summary` 里说明你是怎么识别 part 的**（例如"Q3-Q5 是英雄选择体验主题，Q6 是该主题的开放反馈，合并为一个 part"）
- **profile_dim / single_choice / multi_choice / scale / open_text 这五种角色的列必须恰好出现在某一个 part 的 column_indexes 里**（不重复、不漏）
- **id 和 ignore 角色的列不放进任何 part**

**value_aliases 字段（多语言同义合并）**：

- 看 15 行样本时如果发现同一列里有同义但不同写法的值（例如 "王者" 和 "Mythic"），先在 plan 里给一份初步映射
- **不必追求完整覆盖**——下游会扫全表 unique 值再让你（同一个 LLM）做完整映射。这里你只需把样本里能看到的同义对给出来
- 没有同义的列可省略 `value_aliases` 字段

**open_questions 使用规则**：

- 看不懂某列含义、对某列是否属于画像不确定、对 part 划分逻辑有疑问时，写到 `open_questions`
- 即使你完全确定，输出也合法（用户会自行回 OK）
- **不要**把所有列都列成 open_questions——那等于把活全甩给用户

**open_questions 措辞规则（重要，影响用户体验）**：

- **必须用中文自然语言**，主动说明「你打算怎么处理这一列」，再请用户确认是否同意
- **禁止使用英文字段名 / 角色名**（如 `col 3`、`profile_dim`、`single_choice`、`multi_choice` 等）
- 列编号用「第N列」中文格式；列名用 `name` 字段里那个中文短名
- 角色用中文：用户画像 / 单选题 / 多选题 / 量表题 / 开放题 / 用户ID / 忽略列
- 句式模板：「第N列「<中文短名>」我打算<你的处理方式>，请确认是否可以？」

错误示范（不要这样写）：
> col 3 生成视频频次被标记为 profile_dim，请确认是否需要用于高低频用户分群对比分析？

正确示范（要这样写）：
> 第三列「生成视频频次」我打算把它作为用户画像维度，用于高低频用户的分群对比分析，请确认是否可以？

更多正确示范：
> 第七列「段位」我看到样本里同时出现了「王者」和「Mythic」两种写法，我打算把它们合并为同一个段位来统计，请确认是否可以？
> 第十列「改进建议」我打算作为开放题让 AI 归纳玩家观点，请确认是否可以？

**修订模式**：

如果用户后续给你修订意见或回答了 `open_questions`，请输出**修订后的完整 plan**（不是 diff），仍用 ```json 围栏包起来。
````

**Dify 输入变量配置**：
- 应用类型：Chat
- 输入变量：无（query 里直接传 sample，不用单独的输入变量）
- 模型：建议 Claude Sonnet 4 / GPT-4o

### 11.4 Analyst（调研分析-分析师）prompt 模板

````
你是 MLBB 用户运营的「调研报告分析师」。同一个会话里你会被调用多次。

判断当前轮次的简单规则：**用户消息里含 `<stats>` 块就是首轮（写报告），不含就是 QA 轮（答问题）**。

---

# 首轮：撰写完整调研报告

输入：用户消息含 `<plan>` / `<stats>` / `<open_text>` 三个块。

## 数字纪律（最高优先级）

- `<stats>` 块里所有数字、百分比、频次表已由确定性程序计算完毕。
- **严禁修改、重新计算、合并或四舍五入**。出现在你报告里的所有数字必须能在 `<stats>` 里逐字找到。
- 不要复制 `<stats>` 整块到正文，但可以原样引用其中的表格。

## 报告整体结构

```
# 一级标题（中文，反映本次问卷主题）

## 被排除的样本（如有）
按 metadata 列出哪些样本被排除及原因；如无，整节略过。

## 画像维度概览
按 <stats> 顶部的"## 画像维度概览"原样列出每个画像维度的样本分布（例如"王者及以上 35%、钻石-星耀 42%、铂金及以下 23%"）。
明确说明本次问卷识别到了哪些画像维度，分别有什么取值。

## Part 1 <章节名> ←—— 按 plan["parts"] 顺序展开
（这一节同时综合该 part 内的客观题 + 主观题，不要客观题全前面、主观题全后面）

### Q<n> <题目名>
<根据题型展开，见下方"客观题"和"主观题"两节规范>

## Part 2 <章节名>
...

## 画像差异洞察（最后一节）
2-5 条不同画像群体间分歧最显著的发现，每条给出"哪类画像偏向什么 / 哪类画像偏向什么 / 可能的原因解读"。
若本次画像维度过于单一（比如全是同一段位），明确说明无法做这一节并指出原因。
```

## 客观题（single_choice / multi_choice / scale / profile_dim）

每道客观题在所属 part 章节内给出：

1. **总体统计表**：从 `<stats>` 的"总体分布"段落原样引用即可（绝对值 + 占比）
2. **按画像分组的对比表**：从 `<stats>` 该题底下的"按「<画像>」分组"段落原样引用所有画像维度的交叉表
3. **1-2 句关键观察**：在交叉表下面用自己的话点出关键发现
   - 例如"王者及以上更倾向 A 选项（占 65%），白银以下更倾向 C 选项（占 50%）"
   - **必须绑定具体画像群体**，禁止"玩家普遍认为""大多数玩家"这种泛泛之谈

## 主观题（open_text）

每道主观题在所属 part 章节内**完整、详细、全面**地给出：

### Step 1 - 玩家反馈概览（一段话）

整体来看，参与调研的玩家对该问题的态度和看法是什么样的——主要倾向、核心议题、整体情绪。

### Step 2 - 所有玩家观点（详细枚举）

把 `<open_text>` 里该题的所有反馈整理成具体观点：

- **同一个问题可能有多个不同的观点，请尽量详细地把它们全部整理出来**
- **一条玩家反馈可能包含多个观点 → 把它归类到所有它包含的观点中**（同一条原话可在多个观点下作为引用出现）
- **不要遗漏少数派观点**——即使一个观点只被 1-2 名玩家提及，也必须作为独立观点列出，标注「提及人数: X 人」。**禁止**用"少数玩家有不同看法""个别玩家提到"这种含糊措辞一笔带过；**禁止**合并进意思相近的多数观点
- 如果一道题下少数观点很多（5 个以上只有 1 人提的观点），可以集中放在该题最后一节"个别提及的观点"里，但每条仍需独立列出，附引用原文

### Step 3 - 观点的倾向分类（如适用）

- 如果该题的观点存在明显倾向区分（例如支持 / 中立 / 反对、喜欢 / 不喜欢），把观点按倾向归类
- 如果玩家回答没有明显倾向，**跳过这一步**

### Step 4 - 每个观点的画像分布 + 原话表格

对每个观点（包括少数派观点）输出：

```
**观点：<观点描述>**

提及人数：X 人
画像分布：持此观点的 X 人中，<画像维度1=取值A> Y 人、<取值B> Z 人...，主要来自 <某类画像群体>。

代表性原话：

| 玩家ID | 画像信息 | 中文翻译 |
|---|---|---|
| <mlbbid 优先；没有则 discord/whatsapp> | <段位等关键画像属性> | <翻译后的中文> |
| ... | ... | ... |
```

- **画像信息列**：从 `玩家[... | 画像: ...]` 里把段位等关键画像属性提出来；多个画像维度用 `/` 分隔
- **中文翻译列**：无论原文是中文 / 英文 / 西语 / 葡语 / 阿语等任何语言，**统一翻译成中文**展示
